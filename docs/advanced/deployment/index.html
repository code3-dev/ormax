    Deployment Strategies for Ormax Applications¶

    Proper deployment is crucial for ensuring your Ormax-based application runs smoothly in production. This guide covers best practices, strategies, and considerations for deploying applications that use Ormax ORM across various environments.

    Deployment Considerations¶

    Before deploying, consider these key factors:

    Database Compatibility¶

    Ormax supports multiple database backends, but each has specific deployment requirements:

    Database Production Readiness Connection Stability Scaling Characteristics
    PostgreSQL ★★★★★ ★★★★★ Excellent horizontal scaling
    MySQL ★★★★☆ ★★★★☆ Good scaling with proper configuration
    SQLite ★★☆☆☆ ★★☆☆☆ Limited to single-server deployments
    MSSQL ★★★★☆ ★★★★☆ Enterprise scaling capabilities
    Oracle ★★★★☆ ★★★★★ Enterprise-grade scaling
    Aurora ★★★★★ ★★★★★ Excellent cloud scaling

    Environment Configuration¶

    Ensure proper configuration for each environment:

    # Environment-specific configuration
import os

    db = Database("sqlite:///:memory:")

    Database Migration Strategies¶

    Schema Migration with Ormax¶

    Ormax doesn't include built-in migrations, but here's a recommended approach:

    Migration File Structure¶

    migrations/
├── 0001_initial.py
├── 0002_add_pages_field.py
├── 0003_create_authors_table.py
└── __init__.py

    Sample Migration¶

    # migrations/0002_add_pages_field.py
async def up(db):
    """Add pages field to books table"""
    await db.connection.execute(
        "ALTER TABLE books ADD COLUMN pages INTEGER NOT NULL DEFAULT 100"
    )

async def down(db):
    """Remove pages field from books table"""
    await db.connection.execute(
        "ALTER TABLE books DROP COLUMN pages"
    )

    Migration Runner¶

    import os
import importlib
from ormax import Database

async def run_migrations():
    db = Database(os.getenv("DATABASE_URL"))
    await db.connect()

    # Get applied migrations
    try:
        await db.connection.execute(
            "CREATE TABLE IF NOT EXISTS migrations (name TEXT PRIMARY KEY)"
        )
        applied = await db.connection.fetch_all(
            "SELECT name FROM migrations"
        )
        applied = {m["name"] for m in applied}
    except Exception as e:
        print(f"Error checking migrations table: {e}")
        applied = set()

    # Run pending migrations
    migration_dir = "migrations"
    for filename in sorted(os.listdir(migration_dir)):
        if filename.endswith(".py") and not filename.startswith("__"):
            migration_name = filename[:-3]
            if migration_name in applied:
                continue

            print(f"Running migration: {migration_name}")
            try:
                # Import and run migration
                module = importlib.import_module(f"migrations.{migration_name}")
                async with db.transaction():
                    await module.up(db)
                    await db.connection.execute(
                        "INSERT INTO migrations (name) VALUES (?)", 
                        (migration_name,)
                    )
                print(f"Migration {migration_name} applied successfully")
            except Exception as e:
                print(f"Migration {migration_name} failed: {e}")
                raise

    await db.disconnect()

    Blue-Green Deployment with Database¶

    1. Deploy new application version (green) alongside existing (blue)
    2. Configure green to write to both old and new database schemas
    3. Run data migration in background
    4. Switch traffic to green once migration is complete
    5. Decommission blue environment

    Canary Releases with Database Routing¶

    class DatabaseRouter:
    """Route database operations based on traffic percentage"""
    def __init__(self, primary_db, secondary_db, canary_percentage=5):
        self.primary_db = primary_db
        self.secondary_db = secondary_db
        self.canary_percentage = canary_percentage

    async def get_db(self, user_id=None):
        """Determine which database to use"""
        if user_id and self._is_canary_user(user_id):
            return self.secondary_db
        return self.primary_db

    def _is_canary_user(self, user_id):
        """Determine if user should use canary deployment"""
        return hash(user_id) % 100 < self.canary_percentage

    Optimize connection pool settings for production:

    db = Database(
    os.getenv("DATABASE_URL"),
    min_size=10,          # Minimum connections to keep open
    max_size=50,          # Maximum connections in pool
    pool_recycle=1800,    # Recycle connections after 30 minutes

    Connection Pool Sizing Formula¶

    max_connections = (number_of_instances * max_size_per_instance) * safety_factor

# Where safety_factor is typically 1.2-1.5 to account for spikes

from .notification import send_alert

def setup_alerts():
    """Configure alert thresholds and handlers"""
    from ormax.utils import performance_monitor

    # Query time alerts
    performance_monitor.add_alert(
        "query_time",

    Security Considerations¶

    Database Credentials Management¶

    Never hardcode credentials in your application:

    # BAD: Hardcoded credentials
db = Database("postgresql://user:password@localhost:5432/mydb")

# GOOD: Use environment variables
import os
db = Database(os.getenv("DATABASE_URL"))

    Recommended Credential Management¶

    • Use environment variables for containerized applications
    • Use secret management services (AWS Secrets Manager, HashiCorp Vault)
    • Rotate credentials regularly
    • Use IAM roles where possible (AWS RDS IAM authentication)

    SQL Injection Prevention¶

    Ormax automatically parameterizes queries, but be careful with raw queries:

    # BAD: Potential SQL injection
query = f"SELECT * FROM books WHERE title = '{user_input}'"
results = await db.connection.fetch_all(query)

# GOOD: Proper parameterization
query = "SELECT * FROM books WHERE title = ?"
results = await db.connection.fetch_all(query, (user_input,))

    Connection Security¶

    Ensure secure database connections:

db = Database(

# MySQL with SSL

    Horizontal Scaling¶

    Read Replicas¶

    class ReadWriteRouter:
    """Route reads to replicas and writes to primary"""
    def __init__(self, primary_db, replica_dbs):
        self.primary_db = primary_db
        self.replica_dbs = replica_dbs
        self.current_replica = 0

    async def get_read_db(self):
        """Get a read replica database connection"""
        db = self.replica_dbs[self.current_replica]
        self.current_replica = (self.current_replica + 1) % len(self.replica_dbs)
        return db

    async def get_write_db(self):
        """Get the primary database connection"""
        return self.primary_db

# Usage
router = ReadWriteRouter(primary_db, [replica1, replica2, replica3])

async def get_book(book_id):
    db = await router.get_read_db()
    return await Book.objects(db=db).get(id=book_id)

async def create_book(**kwargs):
    db = await router.get_write_db()
    return await Book.create(db=db, **kwargs)

    Database Sharding¶

    Implement sharding for very large datasets:

    class ShardingRouter:
    """Route requests to appropriate shard"""
    def __init__(self, shards):
        self.shards = shards  # {shard_id: Database}

    def get_shard(self, user_id):
        """Determine which shard to use based on user_id"""
        shard_id = user_id % len(self.shards)
        return self.shards[shard_id]

# Usage
shards = {
    0: Database(os.getenv("SHARD_0_URL")),
    1: Database(os.getenv("SHARD_1_URL")),
    2: Database(os.getenv("SHARD_2_URL")),
}
router = ShardingRouter(shards)

async def get_user_books(user_id):
    db = router.get_shard(user_id)
    return await Book.objects(db=db).filter(user_id=user_id).all()

    Lambda with RDS Proxy¶

    # Lambda function with RDS Proxy
import os
import ormax
from ormax.utils import cached_property

class DatabaseManager:
    """Manage database connection in Lambda environment"""
    _instance = None

    @classmethod
    def get_instance(cls):
        if cls._instance is None:
            cls._instance = cls()
        return cls._instance

    @cached_property
    def db(self):
        """Get database connection with RDS Proxy"""
        return ormax.Database(os.getenv("RDS_PROXY_ENDPOINT"))

    async def get_connection(self):
        """Get a connection from the pool"""
        if not hasattr(self, "_connection") or self._connection._current_transaction is None:
            self._connection = await self.db.transaction().__aenter__()
        return self._connection

    async def release_connection(self):
        """Release the connection back to the pool"""
        if hasattr(self, "_connection") and self._connection._current_transaction:
            await self._connection.__aexit__(None, None, None)
            del self._connection

# Usage in Lambda handler
async def lambda_handler(event, context):
    db_manager = DatabaseManager.get_instance()
    try:
        db = await db_manager.get_connection()
        # Process request
        books = await Book.objects(db=db).all()
        return {"books": [b.to_dict() for b in books]}
    finally:
        await db_manager.release_connection()

    Circuit Breaker Pattern¶

    Prevent cascading failures with a circuit breaker:

    class CircuitBreaker:
    """Circuit breaker for database operations"""
    def __init__(self, failure_threshold=5, recovery_timeout=60):
        self.failure_count = 0
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.last_failure_time = 0
        self.open = False

    def call(self, func, *args, **kwargs):
        """Call a function with circuit breaker protection"""
        if self.open:
            # Check if it's time to try again
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.open = False
            else:
                raise Exception("Database circuit is open")

        try:
            result = func(*args, **kwargs)
            self.failure_count = 0
            return result
        except Exception as e:
            self.failure_count += 1
            self.last_failure_time = time.time()
            if self.failure_count >= self.failure_threshold:
                self.open = True
            raise

# Usage
circuit_breaker = CircuitBreaker()

async def safe_get_books():
    return await circuit_breaker.call(lambda: Book.objects().all())

    async def get_books_with_fallback():

    Common Deployment Issues and Solutions¶

    Issue: Connection Leaks¶

    Symptoms: - Gradually increasing connection count - Application becomes unresponsive - "Too many connections" errors

    Solutions: - Ensure all transactions are properly closed - Use context managers for database operations - Implement connection monitoring - Set appropriate pool_recycle value

    Issue: Slow Deployments¶

    Symptoms: - Long deployment times - Extended downtime during releases - Failed migrations

    Solutions: - Optimize migration scripts - Use zero-downtime deployment patterns - Pre-warm database connections - Run migrations during off-peak hours

    Issue: Data Inconsistency After Deployment¶

    Symptoms: - Missing data - Inconsistent application behavior - Foreign key constraint violations

    Solutions: - Implement proper migration strategies - Use transactions for data migrations - Validate data consistency after deployment - Implement data reconciliation processes

    # 1. Configure database based on environment
    db = Database(os.getenv("DATABASE_URL"))

    # 2. Connect to database
    await db.connect()

    # 3. Register models
    db.register_model(Product)
    db.register_model(Category)
    db.register_model(Order)

    # 4. Run migrations
    await run_migrations()

    # 5. Verify database health
    if not await verify_database_health():
        raise Exception("Database health check failed")

    # 6. Pre-warm critical queries
    await prewarm_queries()

    # 7. Start background monitoring
    asyncio.create_task(monitor_database_performance())

    return db

async def verify_database_health():
    """Verify database is healthy and responsive"""
    try:
        # Check basic query
        await Product.objects().limit(1).all()

        # Check connection pool status
        if hasattr(db.connection, 'pool') and hasattr(db.connection.pool, 'freesize'):
            if db.connection.pool.freesize() < 5:
                logger.warning("Low connection pool availability")

        # Check for slow queries
        slow_queries = await db.connection.fetch_all(
            "SELECT * FROM pg_stat_statements WHERE mean_time > 100"
        )
        if slow_queries:
            logger.warning(f"Found {len(slow_queries)} slow queries")

        return True
    except Exception as e:
        logger.error(f"Database health check failed: {e}")
        return False

async def prewarm_queries():
    """Pre-warm critical queries to reduce first-user latency"""
    # Warm up product listing
    asyncio.create_task(Product.objects().limit(50).all())

    # Warm up category tree
    asyncio.create_task(Category.objects().all())

    # Warm up popular products
    asyncio.create_task(Product.objects().order_by("-popularity").limit(20).all())

import os
import asyncio
from ormax import Database
from .models import Post, Comment, User

async def deploy_blog_platform():
    """Deploy blog platform with read replicas and caching"""
    # 1. Configure primary database
    primary_db = Database(os.getenv("PRIMARY_DB_URL"))
    await primary_db.connect()

    # 2. Configure read replicas
    replica_urls = os.getenv("REPLICA_DB_URLS", "").split(",")
    replica_dbs = [Database(url) for url in replica_urls if url]
    for db in replica_dbs:
        await db.connect()

    # 3. Setup router
    from .router import ReadWriteRouter
    db_router = ReadWriteRouter(primary_db, replica_dbs)

    # 4. Register models with router
    for model in [Post, Comment, User]:
        model.set_db_router(db_router)

    # 5. Run migrations on primary
    await run_migrations(primary_db)

    # 6. Setup caching
    from .cache import setup_cache
    await setup_cache()

    # 7. Start monitoring
    asyncio.create_task(monitor_replication_lag(primary_db, replica_dbs))

    return db_router

async def monitor_replication_lag(primary_db, replica_dbs):
    """Monitor replication lag between primary and replicas"""
    while True:
        try:
            # PostgreSQL specific replication monitoring
            primary_position = await primary_db.connection.fetch_val(
                "SELECT pg_current_wal_lsn()"
            )

            for i, replica_db in enumerate(replica_dbs):
                replica_position = await replica_db.connection.fetch_val(
                    "SELECT pg_last_wal_replay_lsn()"
                )
                lag = await primary_db.connection.fetch_val(
                    "SELECT pg_wal_lsn_diff($1, $2)", 
                    (primary_position, replica_position)
                )

                if lag > 1024 * 1024:  # 1MB lag
                    logger.warning(
                        f"Replication lag on replica {i} is {lag} bytes"
                    )

                # Trigger alert if lag is too high
                if lag > 100 * 1024 * 1024:  # 100MB lag
                    send_replication_lag_alert(i, lag)
        except Exception as e:
            logger.error(f"Error monitoring replication lag: {e}")

        # Check every 30 seconds
        await asyncio.sleep(30)

    Implement Blue-Green Deployments¶

    Minimize downtime with blue-green deployments:

    1. Deploy new version to "green" environment
    2. Run database migrations in a backward-compatible way
    3. Test green environment thoroughly
    4. Switch traffic from blue to green
    5. Decommission blue environment

    Automate Everything¶

    Troubleshooting Deployment Issues¶

    Database Connection Failures¶

    Diagnosis: - Check database URL format - Verify network connectivity - Check firewall rules - Test credentials separately

    Solution: - Use connection string validator - Implement connection retry logic - Verify DNS resolution - Check database server status

    Migration Failures¶

    Diagnosis: - Review migration script - Check database schema - Verify permissions - Examine error logs

    Solution: - Implement transactional migrations - Add rollback procedures - Test migrations in staging - Use idempotent migration patterns

    Performance Degradation After Deployment¶

    Diagnosis: - Compare query execution plans - Check for missing indexes - Monitor resource usage - Analyze slow query logs

    Solution: - Roll back problematic changes - Add necessary indexes - Optimize slow queries - Adjust connection pool settings
