    Performance Monitoring in Ormax¶

    Effective performance monitoring is crucial for maintaining a responsive and scalable application. Ormax provides comprehensive tools and techniques to monitor database performance, identify bottlenecks, and optimize your application's database interactions.

    Understanding Performance Metrics¶

    Before diving into monitoring, it's essential to understand key performance metrics for database operations:

    Key Performance Indicators¶

    Metric Description Target Value
    Query Execution Time Time taken to execute a query < 100ms (simple), < 500ms (complex)
    Connection Pool Usage Percentage of available connections in use < 80% consistently
    Transaction Duration Time spent in a transaction As short as possible
    Query Count per Request Number of queries executed per HTTP request Minimize (N+1 queries are a common issue)
    Error Rate Percentage of failed queries < 0.1%

    Built-in Logging System¶

    Ormax includes a robust logging system that provides detailed insights into database operations.

    Configuring Logging¶

    import logging
from ormax.utils import setup_logging

# Configure logging with default settings
logger = setup_logging(level="INFO")

# Or customize logging configuration
logger = logging.getLogger("ormax")
logger.setLevel(logging.DEBUG)
handler = logging.FileHandler("ormax_performance.log")
formatter = logging.Formatter(
    "%(asctime)s - %(name)s - %(levelname)s - %(message)s"
)
handler.setFormatter(formatter)
logger.addHandler(handler)

    Log Levels and Their Meaning¶

    Level When to Use Example Use Case
    DEBUG Development and detailed diagnostics Tracking individual query execution
    INFO General operational information Connection pool status, migrations
    WARNING Potential issues Slow queries, connection timeouts
    ERROR Recoverable errors Query failures, connection issues
    CRITICAL Severe errors Database unavailable, critical failures

2023-10-15 14:30:46 - ormax - DEBUG - Table books created without foreign keys

    Identifying Slow Queries¶

    Ormax can automatically flag slow queries:

    from ormax.utils import performance_monitor

def monitor_slow_queries(func):
    @wraps(func)
    async def wrapper(*args, **kwargs):
        start = datetime.now()
        result = await func(*args, **kwargs)
        duration = (datetime.now() - start).total_seconds()

        # Record for performance analysis
        performance_monitor.record(f"{func.__name__}", duration)

        # Log slow queries
        if duration > 0.5:  # 500ms threshold
            logger.warning(
                f"Slow query in {func.__name__}: {duration:.2f} seconds"
            )
        return result
    return wrapper

@monitor_slow_queries
async def get_complex_data():
    return await Book.objects().filter(pages__gt=300).all()

    Connection Pool Monitoring¶

    Connection pooling is critical for database performance. Ormax provides tools to monitor and tune your connection pool.

    Viewing Connection Pool Status¶

    async def log_connection_pool_status():
    """Log current connection pool status"""
    if hasattr(db, 'connection') and hasattr(db.connection, 'pool'):
        if hasattr(db.connection.pool, 'size'):
            logger.info(
                f"Connection pool: {db.connection.pool.size()} active, "
                f"{db.connection.pool.freesize()} free, "
                f"{db.connection.pool.size() - db.connection.pool.freesize()} used"
            )
        elif hasattr(db.connection.pool, '_maxsize'):
            # Different implementations have different APIs
            logger.info(
                f"Connection pool: max={db.connection.pool._maxsize}, "
                f"current={len(db.connection.pool._queue._get_heap())}"
            )

    Monitoring Connection Usage Over Time¶

    from ormax.utils import Timer, LRUCache

# Cache to track connection usage
connection_usage = LRUCache(maxsize=1000)

async def monitor_connection_usage():
    """Monitor connection usage patterns"""
    while True:
        try:
            if hasattr(db, 'connection') and hasattr(db.connection, 'pool'):
                used = 0
                total = 0

                # Handle different pool implementations
                if hasattr(db.connection.pool, 'size'):
                    total = db.connection.pool.size()
                    used = total - db.connection.pool.freesize()
                elif hasattr(db.connection.pool, '_maxsize'):
                    total = db.connection.pool._maxsize
                    used = total - len(db.connection.pool._queue._get_heap())

                # Record usage
                timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
                connection_usage.put(timestamp, {"used": used, "total": total})

                # Log high usage
                if used > total * 0.8:  # 80% threshold
                    logger.warning(
                        f"High connection usage: {used}/{total} ({used/total:.1%})"
                    )
        except Exception as e:
            logger.error(f"Error monitoring connection pool: {e}")

        # Check every 5 seconds
        await asyncio.sleep(5)

    Connection Pool Tuning Parameters¶

    Parameter Description Default Tuning Guidance
    min_size Minimum connections in pool 5 Increase for consistent high load
    max_size Maximum connections in pool 20 Match to database server limits
    pool_recycle Time before recycling connections (seconds) 3600 Shorter for unstable networks
    command_timeout Query timeout (seconds) 60 Adjust based on query complexity
    connect_timeout Connection timeout (seconds) 10 Adjust for network conditions

    Query Analysis Tools¶

    EXPLAIN Plans¶

    Use the explain() method to understand how your queries execute:

    # Get the execution plan for a query
plan = await Book.objects().filter(pages__gt=300).explain()
print(plan)

# For more detailed analysis
detailed_plan = await Book.objects().filter(pages__gt=300).explain(analyze=True)

    Interpreting EXPLAIN Output¶

    • Seq Scan: Sequential table scan (potentially slow)
    • Index Scan: Using an index (usually faster)
    • Hash Join: Efficient join method
    • Nested Loop: Can be slow with large datasets
    • Sort: Sorting operation (can be expensive)

    Tracking Query Counts¶

    Monitor the number of queries per operation:

    from ormax.utils import setup_logging
import logging

# Configure a special logger for query counting
query_logger = logging.getLogger("ormax.queries")
query_logger.setLevel(logging.INFO)
query_logger.addHandler(logging.FileHandler("query_count.log"))

def count_queries(func):
    """Decorator to count queries in a function"""
    @wraps(func)
    async def wrapper(*args, **kwargs):
        original_level = logging.getLogger("ormax").getEffectiveLevel()
        logging.getLogger("ormax").setLevel(logging.DEBUG)

        # Clear previous logs
        import io
        log_stream = io.StringIO()
        handler = logging.StreamHandler(log_stream)
        logging.getLogger("ormax").addHandler(handler)

        try:
            result = await func(*args, **kwargs)

            # Count queries from logs
            log_content = log_stream.getvalue()
            query_count = log_content.count("Executing query")

            # Log the count
            query_logger.info(
                f"{func.__name__}: {query_count} queries executed"
            )
            return result
        finally:
            logging.getLogger("ormax").setLevel(original_level)
            logging.getLogger("ormax").removeHandler(handler)

    return wrapper

@count_queries
async def process_books():
    # Function that executes multiple queries
    books = await Book.objects().all()
    for book in books:
        author = await book.author  # This would trigger N+1 queries

    Monitoring Dashboard¶

    Create a real-time monitoring dashboard for your database performance:

    from fastapi import APIRouter, Depends
from ormax.utils import performance_monitor

router = APIRouter()

@router.get("/monitoring/db")
async def get_db_monitoring():
    """Get database performance metrics"""
    # Connection pool status
    pool_status = {"max": 0, "used": 0, "free": 0}
    if hasattr(db, 'connection') and hasattr(db.connection, 'pool'):
        if hasattr(db.connection.pool, 'size'):
            pool_status["max"] = db.connection.pool.size()
            pool_status["used"] = pool_status["max"] - db.connection.pool.freesize()
            pool_status["free"] = db.connection.pool.freesize()
        elif hasattr(db.connection.pool, '_maxsize'):
            pool_status["max"] = db.connection.pool._maxsize
            pool_status["used"] = len(db.connection.pool._queue._get_heap())
            pool_status["free"] = pool_status["max"] - pool_status["used"]

    # Query performance stats
    query_stats = {}
    for operation in performance_monitor.metrics:
        stats = performance_monitor.get_stats(operation)
        if stats:
            query_stats[operation] = {
                "count": stats["count"],
                "avg_time": f"{stats['average']:.4f}",
                "max_time": f"{stats['max']:.4f}"
            }

    # Recent slow queries
    slow_queries = []
    for record in logging.getLogger("ormax").handlers[0].logs:
        if "Slow query" in record.getMessage():
            slow_queries.append(record.getMessage())

    return {
        "connection_pool": pool_status,
        "query_performance": query_stats,
        "slow_queries": slow_queries[-10:],  # Last 10 slow queries
        "timestamp": datetime.now().isoformat()
    }

    PostgreSQL Monitoring¶

    PostgreSQL offers advanced monitoring capabilities:

    async def monitor_postgresql():
    """Monitor PostgreSQL-specific metrics"""
    # Track long-running queries
    long_queries = await db.connection.fetch_all("""
        SELECT pid, now() - query_start AS duration, query 
        FROM pg_stat_activity 
        WHERE state = 'active' AND now() - query_start > interval '5 seconds'
        ORDER BY duration DESC
    """)

    if long_queries:
        for query in long_queries:
            logger.warning(
                f"Long-running query (pid={query['pid']}): "
                f"{query['duration']} - {query['query'][:100]}..."
            )

    # Track lock contention
    locks = await db.connection.fetch_all("""
        SELECT 
            blocked_locks.pid AS blocked_pid,
            blocking_locks.pid AS blocking_pid,
            blocked_activity.query AS blocked_query,
            blocking_activity.query AS blocking_query
        FROM pg_catalog.pg_locks blocked_locks
        JOIN pg_catalog.pg_stat_activity blocked_activity 
            ON blocked_activity.pid = blocked_locks.pid
        JOIN pg_catalog.pg_locks blocking_locks 
            ON blocking_locks.locktype = blocked_locks.locktype
            AND blocking_locks.DATABASE IS NOT DISTINCT FROM blocked_locks.DATABASE
            AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
            AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
            AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
            AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
            AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
            AND blocking_locks.classid IS NOT DISTINCT FROM blocked_locks.classid
            AND blocking_locks.objid IS NOT DISTINCT FROM blocked_locks.objid
            AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
            AND blocking_locks.pid != blocked_locks.pid
        JOIN pg_catalog.pg_stat_activity blocking_activity 
            ON blocking_activity.pid = blocking_locks.pid
        WHERE NOT blocked_locks.GRANTED
    """)

    if locks:
        for lock in locks:
            logger.error(
                f"Deadlock detected: PID {lock['blocking_pid']} blocking "
                f"PID {lock['blocked_pid']}"
            )

    MySQL Monitoring¶

    MySQL-specific monitoring queries:

    async def monitor_mysql():
    """Monitor MySQL-specific metrics"""
    # Track slow queries
    slow_queries = await db.connection.fetch_all("""
        SELECT 
            id, time, info 
        FROM information_schema.processlist 
        WHERE time > 5
        ORDER BY time DESC
    """)

    if slow_queries:
        for query in slow_queries:
            logger.warning(
                f"Slow MySQL query (id={query['id']}): "
                f"{query['time']}s - {query['info'][:100]}..."
            )

    # Track connection usage
    connections = await db.connection.fetch_one("""
        SHOW STATUS LIKE 'Threads_connected'
    """)
    max_connections = await db.connection.fetch_one("""
        SHOW VARIABLES LIKE 'max_connections'
    """)

    if connections and max_connections:
        usage = int(connections['Value']) / int(max_connections['Value'])
        if usage > 0.8:
            logger.warning(
                f"High MySQL connection usage: "
                f"{connections['Value']}/{max_connections['Value']} ({usage:.1%})"
            )

    async def monitor_sqlite():
    """Monitor SQLite-specific metrics"""
    # Track database size
    try:
        import os
        db_path = db.connection_string.replace("sqlite:///", "")
        size = os.path.getsize(db_path)

    except Exception as e:
        logger.error(f"Error monitoring SQLite: {e}")

    Performance Analysis Techniques¶

    Identifying N+1 Query Problems¶

    N+1 queries are a common performance issue where a single query triggers N additional queries:

    async def detect_n_plus_one():
    """Detect potential N+1 query patterns"""
    # Track query patterns
    query_patterns = {}

    # Configure logger to capture queries
    log_stream = io.StringIO()
    handler = logging.StreamHandler(log_stream)
    logger = logging.getLogger("ormax")
    logger.addHandler(handler)

    try:
        # Run common operations
        books = await Book.objects().all()
        for book in books:
            await book.author  # This would trigger N+1 queries

        # Analyze logs
        log_content = log_stream.getvalue()
        queries = [line for line in log_content.split('\n') 
                  if "Executing query" in line]

        # Look for similar queries with different parameters
        pattern_counts = {}
        for query in queries:
            # Extract the base query without parameters
            base_query = re.sub(r'\?.*?FROM', ' ? FROM', query)
            base_query = re.sub(r'WHERE .*? = ', 'WHERE ? = ', base_query)

            pattern_counts[base_query] = pattern_counts.get(base_query, 0) + 1

        # Flag potential N+1 queries
        for pattern, count in pattern_counts.items():
            if count > 10:  # Threshold for potential N+1
                logger.warning(
                    f"Potential N+1 query pattern detected ({count} occurrences): {pattern}"
                )
    finally:
        logger.removeHandler(handler)

    Memory Usage Monitoring¶

    Track memory usage related to database operations:

    import tracemalloc
from ormax.utils import Timer

async def monitor_memory_usage():
    """Monitor memory usage of database operations"""
    tracemalloc.start()

    # Take initial snapshot
    initial_snapshot = tracemalloc.take_snapshot()

    # Run operation to monitor
    with Timer("Memory Test"):
        books = await Book.objects().all()
        # Process books to simulate real usage
        processed = [book.title.upper() for book in books]

    # Take final snapshot
    final_snapshot = tracemalloc.take_snapshot()

    # Compare snapshots
    top_stats = final_snapshot.compare_to(initial_snapshot, 'lineno')

    # Log top memory consumers
    for stat in top_stats[:5]:
        logger.info(f"Memory usage: {stat}")

    # Check for memory leaks
    if len(top_stats) > 0:
        largest_diff = top_stats[0]
        if largest_diff.size_diff > 1024 * 1024:  # 1MB threshold
            logger.warning(
                f"Potential memory issue: {largest_diff.size_diff / 1024:.2f} KB "
                f"in {largest_diff.traceback.format()}"
            )

    tracemalloc.stop()

    Alerting and Notification Systems¶

    Setting Up Performance Alerts¶

    Create alerts for critical performance issues:

    from ormax.utils import Timer
import smtplib
from email.mime.text import MIMEText

class PerformanceMonitor:
    """Monitor performance and send alerts when thresholds are exceeded"""

    def __init__(self):
        self.thresholds = {
            "query_time": 1.0,  # 1 second
            "connection_usage": 0.9,  # 90%
            "error_rate": 0.05  # 5%
        }
        self.alert_history = set()

    def check_query_time(self, func_name, duration):
        """Check if query time exceeds threshold"""
        if duration > self.thresholds["query_time"]:
            alert_key = f"query_time_{func_name}"
            if alert_key not in self.alert_history:
                self.send_alert(
                    "High Query Time",
                    f"Query {func_name} took {duration:.2f}s (threshold: {self.thresholds['query_time']}s)"
                )
                self.alert_history.add(alert_key)
                # Reset alert after 5 minutes
                asyncio.create_task(self.reset_alert(alert_key, 300))

    def check_connection_usage(self, used, total):
        """Check if connection usage exceeds threshold"""
        usage = used / total if total > 0 else 0
        if usage > self.thresholds["connection_usage"]:
            alert_key = "connection_usage"
            if alert_key not in self.alert_history:
                self.send_alert(
                    "High Connection Usage",
                    f"Database connection usage: {usage:.1%} (threshold: {self.thresholds['connection_usage']:.0%})"
                )
                self.alert_history.add(alert_key)
                asyncio.create_task(self.reset_alert(alert_key, 300))

    async def reset_alert(self, alert_key, delay):
        """Reset alert after specified delay"""
        await asyncio.sleep(delay)
        self.alert_history.discard(alert_key)

    def send_alert(self, subject, message):
        """Send alert via email"""
        try:
            msg = MIMEText(message)
            msg['Subject'] = f"[ALERT] {subject}"
            msg['From'] = "monitoring@yourapp.com"
            msg['To'] = "admin@yourapp.com"

            with smtplib.SMTP('localhost') as server:
                server.send_message(msg)
            logger.info(f"Sent alert: {subject}")
        except Exception as e:
            logger.error(f"Failed to send alert: {e}")

# Usage
monitor = PerformanceMonitor()

@measure_time_async
async def get_popular_books():
    result = await Book.objects().filter(pages__gt=300).all()
    # Check performance after execution
    if hasattr(get_popular_books, 'execution_time'):
        monitor.check_query_time("get_popular_books", get_popular_books.execution_time)
    return result

    Performance Optimization Checklist¶

    Use this checklist to maintain optimal performance:

    Daily Checks¶

    • [ ] Review slow query logs
    • [ ] Check connection pool usage
    • [ ] Verify no new N+1 query patterns
    • [ ] Monitor error rates

    Weekly Checks¶

    • [ ] Analyze query execution plans for critical queries
    • [ ] Review index usage and effectiveness
    • [ ] Check for memory usage trends
    • [ ] Validate connection pool settings

    Monthly Checks¶

    • [ ] Benchmark critical operations
    • [ ] Review and optimize expensive queries
    • [ ] Check database statistics and run ANALYZE/VACUUM
    • [ ] Validate monitoring thresholds

    Common Performance Issues and Solutions¶

    Issue: High Connection Usage¶

    Symptoms: - Frequent "Too many connections" errors - Slow response times during peak usage - Connection timeouts

    Solutions: - Increase max_size in connection pool configuration - Ensure transactions are kept short - Implement connection recycling - Identify and fix connection leaks

    Issue: Slow Queries¶

    Symptoms: - Individual queries taking > 500ms - High CPU usage on database server - Slow application response times

    Solutions: - Add appropriate indexes - Optimize query patterns (use select_related/prefetch_related) - Break down complex queries - Consider denormalization for critical paths

    Issue: N+1 Query Problem¶

    Symptoms: - Many similar queries with different parameters - Performance degrades with larger datasets - High query count per request

    Solutions: - Use select_related for single-valued relationships - Use prefetch_related for multi-valued relationships - Consider using values() or values_list() for simpler data needs - Implement batch loading patterns

    Issue: Memory Pressure¶

    Symptoms: - High memory usage during query execution - Process restarts due to OOM (Out of Memory) - Slow garbage collection

    Solutions: - Use pagination for large result sets - Process data in batches - Use only() or defer() to load only necessary fields - Consider streaming results for very large datasets

    Best Practices for Performance Monitoring¶

    Set Realistic Thresholds¶

    • Base thresholds on your specific workload and requirements
    • Start with conservative thresholds and adjust based on historical data
    • Differentiate between development, staging, and production environments

    Monitor in Production¶

    • Development environment performance often differs significantly from production
    • Implement monitoring early in the development cycle
    • Use staging environments to simulate production loads

    Correlate Metrics¶

    • Don't look at metrics in isolation (e.g., query time + connection usage)
    • Correlate database metrics with application metrics
    • Consider the full request lifecycle

    Proactive Monitoring¶

    • Set up alerts for early warning signs, not just critical issues
    • Monitor trends over time to identify gradual degradation
    • Implement synthetic transactions to verify performance

    E-commerce Platform Monitoring¶

    async def monitor_ecommerce_platform():
    """Comprehensive monitoring for e-commerce platform"""
    # Check critical operations
    checks = {
        "product_search": lambda: Product.objects().search("laptop"),
        "cart_processing": lambda: Cart.process_open_carts(),
        "order_creation": lambda: Order.create_sample_order(),
        "inventory_update": lambda: Inventory.update_stock_levels()
    }

    results = {}
    for name, func in checks.items():
        try:
            start = datetime.now()
            await func()
            duration = (datetime.now() - start).total_seconds()
            results[name] = {"status": "OK", "duration": duration}

            # Record for performance monitoring
            performance_monitor.record(name, duration)

            # Alert on slow performance
            if duration > 2.0:  # 2 seconds threshold
                logger.warning(f"Slow {name} operation: {duration:.2f}s")
        except Exception as e:
            results[name] = {"status": "ERROR", "error": str(e)}
            logger.error(f"{name} check failed: {e}")

    # Check database health
    db_health = await check_database_health()

    # Generate summary
    summary = {
        "timestamp": datetime.now().isoformat(),
        "operation_status": results,
        "database_health": db_health,
        "connection_pool": get_connection_pool_status()
    }

    # Log summary
    logger.info(f"Platform health check: {json.dumps(summary)}")

    # Send alerts for critical issues
    if any(status["status"] == "ERROR" for status in results.values()):
        send_critical_alert("E-commerce platform health check failure", summary)

    return summary

    async def monitor_analytics_dashboard():
    """Monitor performance of analytics dashboard queries"""
    # Define critical analytics queries
    analytics_queries = [
        ("monthly_sales", "SELECT * FROM monthly_sales"),
        ("user_engagement", "SELECT * FROM user_engagement"),
        ("product_trends", "SELECT * FROM product_trends"),
        ("geographic_data", "SELECT * FROM geographic_data")
    ]

    results = {}
    for name, query in analytics_queries:
        try:
            start = datetime.now()
            await db.connection.execute(query)
            duration = (datetime.now() - start).total_seconds()
            results[name] = {"status": "OK", "duration": duration}

            # Record for performance monitoring
            performance_monitor.record(f"analytics.{name}", duration)

            # Alert on slow performance
            if duration > 5.0:  # 5 seconds threshold for analytics
                logger.warning(f"Slow analytics query {name}: {duration:.2f}s")
        except Exception as e:
            results[name] = {"status": "ERROR", "error": str(e)}
            logger.error(f"Analytics query {name} failed: {e}")

    # Check for data freshness
    last_update = await db.connection.fetch_one(
        "SELECT MAX(updated_at) as last_update FROM analytics_data"
    )
    if last_update and last_update["last_update"]:
        age = (datetime.now() - last_update["last_update"]).total_seconds()
        if age > 3600:  # 1 hour
            logger.warning(f"Analytics data is {age/60:.1f} minutes old")

    return {
        "timestamp": datetime.now().isoformat(),
        "query_performance": results,
        "data_freshness": last_update["last_update"] if last_update else None
    }

    Troubleshooting Performance Issues¶

    Step-by-Step Diagnosis¶

    When facing performance issues, follow this systematic approach:

    1. Identify the symptom
    2. Is it slow response times?
    3. High error rates?

    4. Resource exhaustion?

    5. Narrow down the scope

    6. Is it affecting all operations or specific ones?
    7. Is it database-specific or application-wide?

    8. When did the issue start?

    9. Collect evidence

    10. Query logs
    11. Connection pool statistics
    12. System resource usage (CPU, memory, I/O)

    13. Database-specific metrics

    14. Formulate hypotheses

    15. Missing index?
    16. Connection leak?

    17. Inefficient query pattern?

    18. Test and validate

    19. Implement potential fixes
    20. Measure impact
    21. Verify resolution

    Common Diagnostic Commands¶

    PostgreSQL¶

    -- Long-running queries
SELECT pid, now() - query_start AS duration, query 
FROM pg_stat_activity 
WHERE state = 'active' AND now() - query_start > interval '5 seconds';

-- Lock contention
SELECT * FROM pg_locks WHERE granted = false;

-- Index usage
SELECT * FROM pg_stat_user_indexes;

    MySQL¶

    -- Slow queries
SHOW FULL PROCESSLIST;

-- Connection usage
SHOW STATUS LIKE 'Threads_connected';
SHOW VARIABLES LIKE 'max_connections';

-- Query execution plans
EXPLAIN SELECT * FROM books WHERE pages > 300;
