    Testing Strategies for Ormax Applications¶

    Thorough testing is essential for building reliable applications with Ormax ORM. This guide covers comprehensive testing strategies, patterns, and tools specifically designed for Ormax-based applications.

    Testing Philosophy with Ormax¶

    Testing Ormax applications requires a layered approach that addresses:

    • Unit tests: Isolated tests of model methods and business logic
    • Integration tests: Tests of database interactions and queries
    • End-to-end tests: Tests of complete application workflows
    • Performance tests: Tests of query efficiency and scalability

    Key Testing Principles¶

    1. Test behavior, not implementation: Focus on what your code does, not how it does it
    2. Test one thing at a time: Each test should verify a single behavior
    3. Keep tests independent: Tests should not depend on each other
    4. Use appropriate test doubles: Choose between mocks, stubs, and fakes based on context
    5. Test edge cases: Don't just test the happy path

    Setting Up a Testing Environment¶

    Test Database Configuration¶

    Always use a separate database for testing:

    # test_config.py
import os
from ormax import Database

def get_test_db():
    """Get a database connection for testing"""
    # Use in-memory SQLite for fast tests
    if os.getenv("TEST_DB") == "sqlite":
        return Database("sqlite:///:memory:")

    # Use dedicated test database for PostgreSQL
    elif os.getenv("TEST_DB") == "postgres":
        return Database("postgresql://test_user:test_password@localhost:5432/test_db")

    # Default to SQLite in-memory
    return Database("sqlite:///:memory:")

    Proper setup and cleanup is critical for reliable tests:

    import pytest

    db.register_model(Author)

        await db.drop_tables()
        await db.disconnect()

    Testing Transactions¶

    @pytest.mark.asyncio
async def test_transaction_rollback(db):
    """Test transaction rollback behavior"""
    # Track if transaction was entered
    transaction_entered = False

    try:
        async with db.transaction():
            transaction_entered = True
            await Book.create(title="Transaction Book", pages=200)
            raise ValueError("Force rollback")
    except ValueError:
        pass

    # Verify transaction rolled back
    assert transaction_entered is True
    books = await Book.objects().all()
    assert len(books) == 0  # Should be empty after rollback

@pytest.mark.asyncio
async def test_nested_transactions(db):
    """Test nested transaction behavior"""
    async with db.transaction():
        await Book.create(title="Outer Book", pages=100)

        try:
            async with db.transaction():
                await Book.create(title="Inner Book", pages=200)
                raise ValueError("Force inner rollback")
        except ValueError:
            pass

        # Outer transaction should still succeed
        await Book.create(title="Another Outer Book", pages=300)

    # Verify results
    books = await Book.objects().all()
    assert len(books) == 2
    assert books[0].title == "Outer Book"
    assert books[1].title == "Another Outer Book"

    Using a Test Database Adapter¶

    Create a lightweight test adapter for faster tests:

    from ormax.connection import DatabaseConnection

class TestConnection(DatabaseConnection):
    """In-memory database connection for testing"""
    def __init__(self):
        super().__init__("test://")
        self.data = {}
        self.queries = []
        self._placeholder = "?"

    async def connect(self):
        pass

    async def disconnect(self):
        pass

    async def execute(self, query, params=None):
        self.queries.append((query, params))
        if query.strip().upper().startswith("INSERT"):
            return 1  # Simulate lastrowid
        return 1  # Simulate rowcount

    async def fetch_one(self, query, params=None):
        self.queries.append((query, params))
        # Simplified query parsing for testing
        if "WHERE" in query:
            where_clause = query.split("WHERE")[1].strip()
            field, value = where_clause.split("=")
            field = field.strip()
            value = value.strip().strip("'")
            # Find matching record
            for table_data in self.data.values():
                for record in table_data:
                    if str(record.get(field)) == value:
                        return record
        return None

    async def fetch_all(self, query, params=None):
        self.queries.append((query, params))
        # Return all data for the first table mentioned
        table_name = query.split()[3]  # Crude parsing for testing
        return self.data.get(table_name, [])

    def get_placeholder(self) -> str:
        return self._placeholder

# Usage in tests
@pytest.fixture

    Testing Signals¶

    import pytest
from ormax.signals import pre_save, post_save

@pytest.mark.asyncio
async def test_book_signals(db):
    """Test signal functionality"""
    # Track signal calls
    pre_save_called = False
    post_save_called = False
    created_value = False

    # Connect to signals
    @pre_save.connect
    async def book_pre_save(sender, instance, **kwargs):
        nonlocal pre_save_called
        pre_save_called = True

    @post_save.connect
    async def book_post_save(sender, instance, created, **kwargs):
        nonlocal post_save_called, created_value
        post_save_called = True
        created_value = created

    # Create a book (should trigger signals)
    book = await Book.create(title="Signal Test", pages=100)

    # Verify signals were called
    assert pre_save_called is True
    assert post_save_called is True
    assert created_value is True

    # Update the book (should trigger post_save with created=False)
    pre_save_called = False
    post_save_called = False
    book.pages = 200
    await book.save()

    # Verify signals were called
    assert pre_save_called is True
    assert post_save_called is True
    assert created_value is False

    assert Timer.get_duration("Complex query") < 0.2  # 200ms target

    Testing N+1 Query Prevention¶

    import pytest
from io import StringIO
import logging

@pytest.mark.asyncio
async def test_n_plus_one_prevention(db):
    """Test N+1 query prevention techniques"""
    # Setup test data
    author = await Author.create(name="Test Author")
    for i in range(10):
        await Book.create(title=f"Book {i}", pages=100, author=author)

    # Configure logger to capture queries
    log_stream = StringIO()
    handler = logging.StreamHandler(log_stream)
    logger = logging.getLogger("ormax")
    logger.addHandler(handler)

    try:
        # Test without select_related (should trigger N+1)
        books = await Book.objects().all()
        for book in books:
            await book.author  # This would trigger N+1 queries

        # Count queries
        log_content = log_stream.getvalue()
        query_count = log_content.count("Executing query")

        # Should have more than 10 queries (1 + 10)
        assert query_count > 10

        # Reset logger
        log_stream = StringIO()

        # Test with select_related (should prevent N+1)
        books = await Book.objects().select_related("author").all()
        for book in books:
            await book.author  # No additional query

        # Count queries
        log_content = log_stream.getvalue()
        query_count = log_content.count("Executing query")

        # Should have only 1 query
        assert query_count == 1
    finally:
        logger.removeHandler(handler)

    Testing Migration Scripts¶

    import pytest
from app.migrations import run_migrations, get_applied_migrations

@pytest.mark.asyncio
async def test_migration_application(db):
    """Test migration application process"""
    # Clear applied migrations
    async with db.transaction():
        await db.connection.execute("DELETE FROM migrations")

    # Run migrations
    await run_migrations()

    # Verify migrations were applied
    applied = await get_applied_migrations()
    assert len(applied) > 0

    # Verify schema changes
    # Check for expected columns
    columns = await db.connection.fetch_all(
        "PRAGMA table_info(books)"
    )
    column_names = [col["name"] for col in columns]
    assert "pages" in column_names
    assert "published_date" in column_names

@pytest.mark.asyncio
async def test_migration_rollback(db):
    """Test migration rollback process"""
    # Run migrations
    await run_migrations()

    # Get the last migration
    applied = await get_applied_migrations()
    last_migration = applied[-1]

    # Create a rollback script
    rollback_script = f"""
    async def down(db):
        await db.connection.execute("ALTER TABLE books DROP COLUMN pages")
    """

    # Save rollback script
    with open(f"migrations/{last_migration}_rollback.py", "w") as f:
        f.write(rollback_script)

    # Run rollback
    from app.migrations import run_rollback
    await run_rollback(last_migration)

    # Verify column was dropped
    columns = await db.connection.fetch_all(
        "PRAGMA table_info(books)"
    )
    column_names = [col["name"] for col in columns]
    assert "pages" not in column_names

    Parameterized Database Tests¶

    import pytest
from app.models import Book
from app.config import get_db

# List of database configurations to test
DB_BACKENDS = [
    {"name": "SQLite", "url": "sqlite:///:memory:"},
    {"name": "PostgreSQL", "url": "postgresql://test:test@localhost:5432/test"},
]

@pytest.mark.parametrize("db_config", DB_BACKENDS, ids=lambda x: x["name"])
@pytest.mark.asyncio
async def test_cross_database(db_config):
    """Test functionality across different database backends"""
    # Setup database
    db = get_db(db_config["url"])
    await db.connect()
    db.register_model(Book)
    await db.create_tables()

    try:
        # Run identical tests on each database
        await Book.create(title="Cross-DB Test", pages=200)
        book = await Book.objects().get(title="Cross-DB Test")
        assert book.pages == 200

        # Test database-specific features
        if "sqlite" in db_config["url"]:
            # SQLite-specific test
            result = await db.connection.fetch_one(
                "PRAGMA foreign_keys"
            )
            assert result["foreign_keys"] == 1
        elif "postgresql" in db_config["url"]:
            # PostgreSQL-specific test
            result = await db.connection.fetch_one(
                "SHOW server_version"
            )
            assert "PostgreSQL" in result["server_version"]
    finally:
        await db.drop_tables()
        await db.disconnect()

    Testing Anti-Patterns to Avoid¶

    Avoid Testing Implementation Details¶

    BAD: Testing specific SQL queries

    # BAD: Testing specific SQL generation
def test_book_query():
    qs = Book.objects().filter(title="Test")
    assert "WHERE title = ?" in qs._build_select_query()

    GOOD: Testing behavior instead

    # GOOD: Testing behavior
@pytest.mark.asyncio
async def test_book_filtering(db):
    await Book.create(title="Test Book", pages=100)
    books = await Book.objects().filter(title="Test Book").all()
    assert len(books) == 1

    Avoid Over-Mocking¶

    BAD: Mocking too much of the ORM

    # BAD: Over-mocking
def test_book_creation():
    with patch("ormax.Model.save", return_value=None):
        with patch("ormax.Model.__init__", return_value=None):
            book = Book(title="Test", pages=100)
            assert book.title == "Test"

    GOOD: Testing with a real database connection

    # GOOD: Using a real test database
@pytest.mark.asyncio
async def test_book_creation(db):
    book = await Book.create(title="Test", pages=100)
    assert book.id is not None
    assert book.title == "Test"

    Avoid Slow Tests¶

    BAD: Creating large datasets in every test

    # BAD: Creating 10,000 records in every test
@pytest.mark.asyncio
async def test_book_search(db):
    # Creates 10,000 records - too slow for unit test
    for i in range(10000):
        await Book.create(title=f"Book {i}", pages=i)
    # Test logic...

    GOOD: Using minimal test data

    # GOOD: Using minimal test data
@pytest.mark.asyncio
async def test_book_search(db):
    # Create just enough data for the test
    await Book.create(title="Searchable Book", pages=100)
    await Book.create(title="Another Book", pages=200)

    # Test search
    results = await Book.objects().filter(title__contains="Searchable").all()
    assert len(results) == 1

    Testing Pyramid for Ormax Applications¶

    A well-structured testing strategy follows the testing pyramid:

              Performance Tests (Few)
                /    \
               /      \
Integration Tests (Medium)
             |
             |
        Unit Tests (Many)

    Unit Tests (70% of tests)¶

    • Test individual model methods
    • Test validation logic
    • Test simple business rules
    • Fast, isolated, no database required (or in-memory)

    Integration Tests (25% of tests)¶

    • Test database queries
    • Test relationships
    • Test transactions
    • Test with real database connection

    Performance Tests (5% of tests)¶

    • Test query performance
    • Test bulk operations
    • Test under load
    • Measure execution time and resource usage

from decimal import Decimal
from app.models import Product, Category, Inventory

async def test_product_inventory_management(db):

    Troubleshooting Common Testing Issues¶

    Issue: Tests Fail with Database Connection Errors¶

    Symptoms: - DatabaseError: Failed to connect to database - Connection timeout errors - Authentication failures

    Solutions: - Verify test database is running - Check database credentials in test configuration - Ensure database container is ready before tests start - Add retry logic for database connections

    Issue: Tests Fail Due to Foreign Key Constraints¶

    Symptoms: - IntegrityError: FOREIGN KEY constraint failed - Tests fail when deleting objects

    Solutions: - Use transactions that roll back after each test - Delete objects in the correct order (child before parent) - Disable foreign key checks during teardown (database-specific)

    Issue: Tests Are Too Slow¶

    Symptoms: - Long test execution time - CI pipeline takes too long - Developers avoid running tests

    Solutions: - Use in-memory SQLite for unit tests - Minimize test data size - Avoid heavy setup in fixtures - Use mocking for external dependencies - Run tests in parallel

    Issue: Flaky Tests¶

    Symptoms: - Tests pass sometimes and fail other times - Failures seem random - Hard to reproduce locally

    Solutions: - Check for race conditions in async code - Ensure proper isolation between tests - Avoid shared state between tests - Add proper cleanup in teardown - Increase timeouts for slow operations
