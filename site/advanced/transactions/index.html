    Transaction Management in Ormax¶

    Transactions are essential for maintaining data integrity in database applications. Ormax provides robust transaction management with support for nested transactions, savepoints, and multiple database backends.

    Understanding Transactions¶

    A database transaction is a sequence of operations performed as a single logical unit of work. Transactions ensure that either all operations succeed or none do, maintaining database consistency.

    ACID Properties¶

    Ormax transactions maintain the ACID properties:

    • Atomicity: All operations in a transaction succeed or fail together
    • Consistency: Transactions bring the database from one valid state to another
    • Isolation: Concurrent transactions don't interfere with each other
    • Durability: Once committed, changes persist even after system failure

    Basic Transaction Usage¶

    The simplest way to use transactions in Ormax is with the transaction() context manager:

    async with db.transaction():
    # All operations within this block are part of a transaction
    book = await Book.create(title="Transaction Book", pages=200)
    author = await Author.create(name="Transaction Author")

    # If an exception occurs, the transaction will be rolled back automatically

    Why Use Transactions?¶

    Without transactions:

    # Without transaction
book = await Book.create(title="Book 1", pages=200)
# If exception occurs here, book is created but author isn't
author = await Author.create(name="Author 1")

    With transactions:

    async with db.transaction():
    book = await Book.create(title="Book 1", pages=200)
    author = await Author.create(name="Author 1")
    # Either both succeed or both fail

    Nested Transactions with Savepoints¶

    Ormax supports nested transactions using savepoints, which is particularly useful for complex business logic.

    How Nested Transactions Work¶

    async with db.transaction():
    # Outer transaction
    await Book.create(title="Outer Book", pages=100)

    try:
        async with db.transaction():
            # Inner transaction (savepoint)
            await Book.create(title="Inner Book", pages=200)
            # This will roll back only the inner transaction
            raise ValueError("Something went wrong")
    except ValueError:
        pass

    # Outer transaction still succeeds
    await Book.create(title="Another Outer Book", pages=300)

    Savepoint Behavior by Database¶

    Database Nested Transaction Support Savepoint Syntax
    PostgreSQL Full support SAVEPOINT name, RELEASE SAVEPOINT name, ROLLBACK TO name
    MySQL Full support SAVEPOINT name, RELEASE SAVEPOINT name, ROLLBACK TO name
    SQLite Full support SAVEPOINT name, RELEASE name, ROLLBACK TO name
    MSSQL Full support SAVE TRANSACTION name, ROLLBACK TRANSACTION name
    Oracle Limited support SAVEPOINT name, ROLLBACK TO name
    Aurora Full support (MySQL-compatible) Same as MySQL

    Manual Transaction Control¶

    For more complex scenarios, you can manage transactions manually:

    # Start transaction manually
transaction = await db.transaction().__aenter__()

try:
    await Book.create(title="Manual Transaction Book", pages=250)
    # Commit if everything is good
    await transaction.__aexit__(None, None, None)
except Exception as e:
    # Rollback on error
    await transaction.__aexit__(type(e), e, None)
    raise

    Transaction State Checking¶

    You can check if you're currently in a transaction:

    is_in_transaction = db._current_transaction is not None
transaction_level = getattr(db, '_transaction_level', 0)

    Transaction Isolation Levels¶

    Ormax allows you to set transaction isolation levels for specific needs:

    async with db.transaction(isolation_level="SERIALIZABLE"):
    # Highly isolated transaction
    await process_critical_data()

    Supported Isolation Levels¶

    Level Description Use Case
    READ UNCOMMITTED Can see uncommitted changes from other transactions Rarely used, highest performance
    READ COMMITTED Can only see committed changes (default for most databases) General purpose
    REPEATABLE READ Ensures consistent reads within a transaction When consistent reads are critical
    SERIALIZABLE Complete isolation, prevents phantom reads Critical financial operations

    Note: Not all databases support all isolation levels. Ormax will use the closest available level for your database.

    Advanced Transaction Patterns¶

    Transaction with Retry Logic¶

    For handling transient errors like deadlocks:

    from ormax.utils import retry_async

@retry_async(max_attempts=3, delay=0.1, backoff=2.0)
async def create_order_with_retry(user, items):
    async with db.transaction():
        # Create order and process items
        order = await Order.create(user=user)
        for item in items:
            await OrderItem.create(order=order, item=item, quantity=item.quantity)
        return order

    Transaction with Time Limit¶

    Set a timeout for transactions to prevent long-running operations:

    from ormax.utils import timeout_async

@timeout_async(seconds=5)
async def process_payment():
    async with db.transaction():
        # Payment processing logic
        await Payment.process()

    Savepoint-Based Partial Rollbacks¶

    Use savepoints for granular control within a transaction:

    async with db.transaction():
    # Initial operation
    await create_invoice()

    # Create a savepoint
    sp1 = await db.transaction().__aenter__()

    try:
        # Critical operation that might fail
        await process_payment()
    except PaymentError:
        # Roll back just the payment processing
        await sp1.__aexit__(Exception, None, None)
        # Continue with alternative payment method
        await process_alternative_payment()

    # Continue with the rest of the transaction
    await send_confirmation_email()

    Working with Model Transactions¶

    Transaction-Aware Model Methods¶

    Some model methods automatically participate in transactions:

    # These operations will be part of the current transaction
async with db.transaction():
    book = await Book.create(title="Transactional Book", pages=300)
    await book.update(pages=350)
    await book.delete()

    Bulk Operations in Transactions¶

    Perform bulk operations within a single transaction for performance:

    async with db.transaction():
    books_data = [
        {"title": f"Book {i}", "pages": 100 + i} for i in range(1000)
    ]
    await Book.bulk_create(books_data)

    Database-Specific Transaction Features¶

    PostgreSQL Advisory Locks¶

    Use PostgreSQL's advisory locks within transactions:

    async with db.transaction():
    # Acquire an advisory lock
    await db.connection.execute("SELECT pg_advisory_xact_lock(12345)")
    # Perform operations that require exclusive access
    await process_exclusive_operation()

    MySQL Locking Reads¶

    Use MySQL's locking reads within transactions:

    async with db.transaction():
    # Lock rows for update
    books = await Book.objects().filter(pages__gt=300).for_update().all()
    # Now update these books without interference
    for book in books:
        book.pages -= 10
        await book.save()

    Error Handling and Recovery¶

    Handling Transaction Errors¶

    Proper error handling is critical for transaction management:

    try:
    async with db.transaction():
        # Transaction operations
        await process_data()
except DatabaseError as e:
    # Handle specific database errors
    logger.error(f"Database error in transaction: {e}")
    # The transaction is already rolled back
except CustomApplicationError as e:
    # Handle application-specific errors
    logger.error(f"Application error: {e}")
    # The transaction is already rolled back

    Common Transaction Errors¶

    Error Cause Solution
    TransactionRollbackError Deadlock or serialization failure Implement retry logic
    IntegrityError Constraint violation Fix data or adjust constraints
    OperationalError Connection issues Reconnect and retry
    ProgrammingError SQL syntax error Fix the query

    Deadlock Prevention Strategies¶

    1. Access tables in consistent order across your application
    2. Keep transactions short - do only what's necessary
    3. Use appropriate isolation levels - don't use SERIALIZABLE unless needed
    4. Implement retry logic for deadlock-prone operations
    5. Set timeouts to prevent long-running transactions

    Performance Considerations¶

    Transaction Size¶

    • Small transactions: Better concurrency, less locking
    • Large transactions: Fewer round-trips, but more locking

    Best Practice: Find the right balance for your application.

    Batch Processing in Transactions¶

    For large data operations, process in batches within transactions:

    async def process_in_batches(items, batch_size=100):
    for i in range(0, len(items), batch_size):
        batch = items[i:i+batch_size]
        async with db.transaction():
            for item in batch:
                await process_item(item)

    Read-Only Transactions¶

    Mark transactions as read-only when possible for better performance:

    async with db.transaction(read_only=True):
    # Only read operations here
    books = await Book.objects().all()

    Note: PostgreSQL and Oracle support explicit read-only transactions; other databases ignore this setting.

    Real-World Examples¶

    E-commerce Order Processing¶

    async def process_order(user, cart_items):
    """
    Process an e-commerce order with proper transaction management
    """
    async with db.transaction():
        # Create order
        order = await Order.create(
            user=user,
            status="PROCESSING",
            total_amount=calculate_total(cart_items)
        )

        # Process each item
        for item in cart_items:
            # Check inventory
            product = await Product.get(id=item.product_id)
            if product.inventory < item.quantity:
                raise InsufficientInventoryError(
                    f"Not enough {product.name} in stock"
                )

            # Reserve inventory
            product.inventory -= item.quantity
            await product.save()

            # Create order item
            await OrderItem.create(
                order=order,
                product=product,
                quantity=item.quantity,
                price=product.price
            )

        # Update order status
        order.status = "COMPLETED"
        await order.save()

        return order

    Banking Transaction System¶

    async def transfer_funds(from_account, to_account, amount):
    """
    Transfer funds between accounts with transaction safety
    """
    # Validate inputs
    if amount <= 0:
        raise ValueError("Transfer amount must be positive")

    async with db.transaction():
        # Get accounts with row locking to prevent race conditions
        from_acc = await Account.get(id=from_account, for_update=True)
        to_acc = await Account.get(id=to_account, for_update=True)

        # Check sufficient funds
        if from_acc.balance < amount:
            raise InsufficientFundsError(
                f"Account {from_account} has insufficient funds"
            )

        # Perform transfer
        from_acc.balance -= amount
        to_acc.balance += amount

        # Save changes
        await from_acc.save()
        await to_acc.save()

        # Create transaction records
        await Transaction.create(
            account_id=from_account,
            amount=-amount,
            description=f"Transfer to {to_account}"
        )
        await Transaction.create(
            account_id=to_account,
            amount=amount,
            description=f"Transfer from {from_account}"
        )

        return {
            "from_balance": from_acc.balance,
            "to_balance": to_acc.balance
        }

    Unit Testing Transactions¶

    When testing transactional code, use in-memory SQLite for speed:

    import pytest
from unittest.mock import AsyncMock

@pytest.mark.asyncio
async def test_order_processing():
    # Setup test database
    db = Database("sqlite:///:memory:")
    db.register_model(Order)
    db.register_model(OrderItem)
    await db.create_tables()

    # Test transactional function
    user = await User.create(name="Test User")
    cart_items = [{"product_id": 1, "quantity": 2}]

    # Should complete without errors
    order = await process_order(user.id, cart_items)
    assert order.status == "COMPLETED"

    # Verify inventory was updated
    product = await Product.get(id=1)
    assert product.inventory == INITIAL_INVENTORY - 2

    Simulating Transaction Failures¶

    Test your error handling by simulating failures:

    @pytest.mark.asyncio
async def test_order_processing_insufficient_inventory():
    # Setup test database
    db = Database("sqlite:///:memory:")
    # ... setup models and data ...

    # Create product with limited inventory
    product = await Product.create(name="Test Product", inventory=1)

    # Attempt to order more than available
    cart_items = [{"product_id": product.id, "quantity": 2}]

    # Should raise InsufficientInventoryError
    with pytest.raises(InsufficientInventoryError):
        await process_order(user.id, cart_items)

    # Verify no changes were made
    product = await Product.get(id=product.id)
    assert product.inventory == 1  # No change

    Best Practices¶

    Keep Transactions Short¶

    • Do as little work as possible within a transaction
    • Move non-database operations outside transactions
    • Process data before entering the transaction

    Avoid User Interaction in Transactions¶

    • Never wait for user input inside a transaction
    • Complete all business logic before starting a transaction

    Use Appropriate Isolation Levels¶

    • Use the lowest isolation level that meets your needs
    • Don't use SERIALIZABLE unless absolutely necessary

    Implement Retry Logic¶

    • For operations prone to deadlocks or serialization failures
    • Use exponential backoff for retries

    Monitor Long-Running Transactions¶

    • Set up monitoring for transactions exceeding expected duration
    • Implement timeouts to prevent runaway transactions

    Troubleshooting Common Issues¶

    Stale Data After Transaction¶

    Issue: Reading data that appears unchanged after a transaction

    Solution: - Verify you're not in a nested transaction that hasn't committed - Check if you're using the correct isolation level - Ensure you're not caching results outside the transaction

    Deadlocks¶

    Issue: TransactionRollbackError due to deadlocks

    Solution: - Access tables in consistent order across your application - Keep transactions as short as possible - Implement retry logic for deadlock-prone operations - Consider using lower isolation levels

    Transaction Not Rolling Back¶

    Issue: Changes persist after an exception within a transaction

    Solution: - Ensure you're using the async context manager correctly - Verify you're not catching and ignoring exceptions - Check if you're using manual transaction control correctly

    Savepoint Errors in Nested Transactions¶

    Issue: ProgrammingError related to savepoints in nested transactions

    Solution: - Verify your database supports nested transactions - Check if you're properly exiting inner transaction contexts - Ensure you're not mixing manual and context manager transaction styles

    Performance Tuning¶

    Connection Pool Configuration¶

    Optimize connection pool settings for transaction-heavy workloads:

    db = Database(
    "postgresql://user:password@localhost:5432/mydb",
    min_size=10,      # Increase for high transaction volume
    max_size=50,      # Higher for more concurrent transactions
    command_timeout=30  # Adjust based on transaction complexity
)

    Batch Size Optimization¶

    Find the optimal batch size for bulk operations:

    # Experiment with different batch sizes
BATCH_SIZES = [50, 100, 200, 500]

for batch_size in BATCH_SIZES:
    start = time.time()
    async with db.transaction():
        await process_in_batches(items, batch_size)
    duration = time.time() - start
    print(f"Batch size {batch_size}: {duration:.2f} seconds")

    Monitoring Long Transactions¶

    Implement monitoring for long-running transactions:

    from ormax.utils import Timer

async def monitored_transaction():
    with Timer("Order Processing"):
        async with db.transaction():
            # Transaction operations
            await process_order()
