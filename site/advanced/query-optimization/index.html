    Query Optimization in Ormax¶

    Efficient database queries are critical for application performance. This guide covers advanced techniques to optimize your Ormax queries, reduce database load, and improve response times.

    Understanding Query Performance¶

    Before optimizing, it's essential to understand how queries perform and where bottlenecks occur.

    Measuring Query Performance¶

    Ormax provides built-in tools to measure query execution time:

    from ormax.utils import measure_time_async

@measure_time_async
async def get_popular_books():
    return await Book.objects().filter(pages__gt=300).all()

    This decorator will log the execution time of your function, helping you identify slow queries.

    Viewing SQL Queries¶

    To see the actual SQL being executed:

    # Enable query logging
import logging
logging.basicConfig(level=logging.DEBUG)

# Or configure specifically for Ormax
logger = logging.getLogger("ormax")
logger.setLevel(logging.DEBUG)

    With debug logging enabled, Ormax will print all executed queries to the console.

    Understanding Query Execution Plans¶

    Different databases provide tools to analyze query execution:

    # Get the execution plan for a query
plan = await Book.objects().filter(pages__gt=300).explain()
print(plan)

    Note: The explain() method is database-specific and may return different formats depending on your database backend.

    Indexing Strategies¶

    Proper indexing is the most effective way to improve query performance.

    When to Create Indexes¶

    Create indexes for: - Fields used in WHERE clauses - Fields used in ORDER BY - Foreign key fields - Fields used in JOIN conditions - Fields frequently used in search operations

    Creating Indexes in Ormax¶

    You can define indexes in your model's _meta:

    class Book(ormax.Model):
    title = CharField(max_length=200)
    pages = IntegerField()

    _meta = {
        "indexes": [
            {"fields": ["title"], "unique": True},
            {"fields": ["pages"], "name": "pages_idx"},
            {"fields": ["title", "pages"], "name": "title_pages_idx"}
        ]
    }

    Database-Specific Index Types¶

    Different databases support specialized index types:

    class Product(ormax.Model):
    name = CharField(max_length=200)
    description = TextField()

    _meta = {
        "indexes": [
            # PostgreSQL full-text index
            {"fields": ["name", "description"], "type": "GIN", "name": "product_search_idx"}
        ]
    }

    Managing Indexes¶

    Ormax provides methods to manage indexes:

    # Create all indexes for a model
await Book.create_indexes()

# Create a specific index
await Book.create_index("title_idx")

# Drop an index
await Book.drop_index("title_idx")

    Optimizing Query Execution¶

    Using select_related Effectively¶

    select_related follows foreign key relationships in a single query:

    # BAD: N+1 queries
books = await Book.objects().all()
for book in books:
    print(book.author.name)  # Triggers a query for each book

# GOOD: One query with JOIN
books = await Book.objects().select_related("author").all()
for book in books:
    print(book.author.name)  # No additional queries

    Best Practices for select_related¶

    • Use for single-valued relationships (ForeignKey, OneToOne)
    • Don't chain too many levels (2-3 levels max)
    • Be specific about which relationships to follow: 
      books = await Book.objects().select_related("author__publisher").all()

    Using prefetch_related Effectively¶

    prefetch_related fetches related objects in separate queries:

    # BAD: N+1 queries
authors = await Author.objects().all()
for author in authors:
    for book in author.books.all():  # Triggers a query per author
        print(book.title)

# GOOD: Two queries total
authors = await Author.objects().prefetch_related("books").all()
for author in authors:
    for book in author.books.all():  # No additional queries
        print(book.title)

    Best Practices for prefetch_related¶

    • Use for multi-valued relationships (reverse ForeignKey)
    • Combine with filtering: 
      authors = await Author.objects().prefetch_related(
    Prefetch("books", queryset=Book.objects().filter(pages__gt=300))
).all()
    • Avoid nested prefetching unless absolutely necessary

    Using only() and defer()¶

    Load only the fields you need:

    # Load only title and pages
books = await Book.objects().only("title", "pages").all()

# Accessing other fields will return None
for book in books:
    print(book.title)  # Works
    print(book.author)  # Returns None

    # Load all fields except bio
authors = await Author.objects().defer("bio").all()

# Accessing bio will trigger an additional query
for author in authors:
    print(author.name)  # Works
    print(author.bio)   # Triggers additional query

    Best Practices for only() and defer()¶

    • Use only() when you need just a few fields from a model with many fields
    • Use defer() when you need most fields except one or two large ones
    • Never use these methods with create() or update()

    Batch Processing Large Result Sets¶

    For processing large datasets:

    async def process_large_dataset():
    # Process in batches of 100
    async for batch in Book.objects().batch(100):
        for book in batch:
            # Process book
            await process_book(book)

    Or with manual batching:

    async def process_in_batches(queryset, batch_size=100):
    offset = 0
    while True:
        batch = await queryset.limit(batch_size).offset(offset).all()
        if not batch:
            break
        for item in batch:
            # Process item
            await process_item(item)
        offset += batch_size

    Using values() and values_list()¶

    When you only need specific field values:

    # Get titles and pages as dictionaries
book_data = await Book.objects().values("title", "pages").all()

# Get only titles as a flat list
titles = await Book.objects().values_list("title", flat=True).all()

    Performance Benefits¶

    • Reduced memory usage
    • Faster serialization
    • Less network overhead
    • No model instantiation overhead

    Advanced Query Optimization Techniques¶

    Query Caching¶

    Ormax provides caching utilities:

    from ormax.utils import memoize_async

@memoize_async(maxsize=100)
async def get_popular_books():
    return await Book.objects().filter(pages__gt=300).all()

    Cache Invalidation Strategies¶

    • Time-based expiration
    • Manual invalidation on data changes
    • Version-based caching

    Using Raw SQL for Complex Queries¶

    For queries that can't be expressed with the QuerySet API:

    # Complex analytical query
results = await Book.objects().raw(
    """
    SELECT author_id, AVG(pages) as avg_pages
    FROM book
    GROUP BY author_id
    HAVING AVG(pages) > ?
    """,
    (300,)
).execute()

    Optimizing Aggregation Queries¶

    # Efficient count
total = await Book.objects().filter(pages__gt=300).count()

# Optimized aggregation
from ormax.query import Aggregation
avg_pages = await Aggregation.avg(Book.objects().filter(pages__gt=300), "pages")

    Using Database-Specific Features¶

    PostgreSQL JSONB Optimization¶

    # Index JSONB field
class Product(ormax.Model):
    data = JSONField()

    _meta = {
        "indexes": [
            {"fields": ["data"], "type": "GIN", "name": "data_idx"}
        ]
    }

# Query JSONB field efficiently
products = await Product.objects().filter(
    data__contains={"color": "red"}
).all()

    MySQL Full-Text Search¶

    # Create full-text index
class Article(ormax.Model):
    title = CharField(max_length=200)
    content = TextField()

    _meta = {
        "indexes": [
            {"fields": ["title", "content"], "type": "FULLTEXT", "name": "search_idx"}
        ]
    }

# Use MATCH AGAINST for full-text search
articles = await Article.objects().raw(
    "SELECT * FROM article WHERE MATCH(title, content) AGAINST (?)",
    ("search term",)
).execute()

    Common Query Performance Issues¶

    The N+1 Query Problem¶

    Symptoms: - Application makes many similar queries - Performance degrades as dataset grows - Logs show repeated queries

    Solution: - Use select_related for single-valued relationships - Use prefetch_related for multi-valued relationships

    Unindexed Foreign Keys¶

    Symptoms: - Slow JOIN operations - Slow filter operations on foreign key fields

    Solution: - Ensure all foreign key fields are indexed - Consider composite indexes for common query patterns

    Loading Unnecessary Data¶

    Symptoms: - High memory usage - Slow serialization - Large network payloads

    Solution: - Use only() or values() to load only needed fields - Avoid fetching entire related objects when not needed

    Inefficient Aggregation¶

    Symptoms: - Slow count operations on large tables - Slow AVG, SUM, MAX, MIN operations

    Solution: - Use database-level aggregation instead of Python-level - Consider materialized views for frequently used aggregations

    Performance Monitoring¶

    Tracking Query Statistics¶

    Ormax includes a performance monitor:

    from ormax.utils import performance_monitor

# Record query execution time
start = datetime.now()
results = await Book.objects().all()
duration = (datetime.now() - start).total_seconds()
performance_monitor.record("book_query", duration)

# Get statistics
stats = performance_monitor.get_stats("book_query")
print(f"Average query time: {stats['average']:.4f} seconds")

    Monitoring Long-Running Queries¶

    Set up alerts for slow queries:

    from ormax.utils import measure_time_async

def monitor_slow_queries(func):
    @measure_time_async
    async def wrapper(*args, **kwargs):
        result = await func(*args, **kwargs)
        # Log queries taking more than 500ms
        if wrapper.execution_time > 0.5:
            logger.warning(
                f"Slow query in {func.__name__}: {wrapper.execution_time:.2f} seconds"
            )
        return result
    return wrapper

@monitor_slow_queries
async def get_complex_data():
    # Complex query operations
    return await Book.objects().filter(pages__gt=300).all()

    Query Optimization Checklist¶

    Before deploying to production, verify these optimizations:

    • [ ] All foreign key fields have indexes
    • [ ] Frequently filtered fields have indexes
    • [ ] N+1 query issues are resolved with select_related/prefetch_related
    • [ ] Only necessary fields are loaded (using only()/values())
    • [ ] Large datasets are processed in batches
    • [ ] Complex aggregations use database-level operations
    • [ ] Query execution plans have been reviewed
    • [ ] Slow queries have been identified and optimized

    Real-World Optimization Examples¶

    E-commerce Product Listing¶

    async def get_product_list(category_id, page=1, per_page=20):
    """Optimized product listing with proper indexing and field selection"""
    # Start timer for monitoring
    start_time = datetime.now()

    # Get category with minimal data
    category = await Category.objects().only("id", "name").get(id=category_id)

    # Prefetch related data in efficient batches
    products = await Product.objects() \
        .filter(category=category_id) \
        .select_related("brand") \
        .prefetch_related(Prefetch(
            "reviews", 
            queryset=Review.objects().order_by("-created_at").limit(3)
        )) \
        .only("id", "name", "price", "image_url", "brand_id") \
        .order_by("-featured", "-created_at") \
        .limit(per_page) \
        .offset((page-1) * per_page) \
        .all()

    # Get total count efficiently
    total = await Product.objects().filter(category=category_id).count()

    # Record performance
    duration = (datetime.now() - start_time).total_seconds()
    performance_monitor.record("product_list", duration)

    return {
        "category": category,
        "products": products,
        "total": total,
        "page": page,
        "pages": (total + per_page - 1) // per_page
    }

    Analytics Dashboard¶

    async def get_dashboard_analytics():
    """Optimized analytics queries using database-level aggregation"""
    # Use database aggregation for counts
    total_books = await Book.objects().count()
    total_authors = await Author.objects().count()

    # Use raw SQL for complex analytics
    monthly_stats = await Book.objects().raw(
        """
        SELECT 
            DATE_TRUNC('month', published_date) as month,
            COUNT(*) as book_count,
            AVG(pages) as avg_pages
        FROM book
        WHERE published_date >= CURRENT_DATE - INTERVAL '1 year'
        GROUP BY month
        ORDER BY month
        """
    ).execute()

    # Get popular authors with efficient query
    popular_authors = await Author.objects() \
        .annotate(book_count="COUNT(books)") \
        .order_by("-book_count") \
        .limit(10) \
        .values("id", "name", "book_count") \
        .all()

    # Get category distribution
    category_stats = await Book.objects() \
        .values("category__name") \
        .annotate(count="COUNT(*)") \
        .all()

    return {
        "total_books": total_books,
        "total_authors": total_authors,
        "monthly_stats": monthly_stats,
        "popular_authors": popular_authors,
        "category_stats": category_stats
    }

    Database-Specific Optimization Tips¶

    PostgreSQL Optimization¶

    • Use psycopg2.extras.DictCursor for faster result processing
    • Enable connection pooling with appropriate settings
    • Use pg_stat_statements to identify slow queries
    • Consider partial indexes for common filter patterns

    MySQL Optimization¶

    • Use utf8mb4 character set for full Unicode support
    • Configure innodb_buffer_pool_size appropriately
    • Use EXPLAIN to analyze query plans
    • Consider query caching for frequently executed queries

    SQLite Optimization¶

    • Use PRAGMA journal_mode=WAL for better concurrency
    • Use PRAGMA synchronous=NORMAL for better performance
    • Create indexes before inserting large amounts of data
    • Use transactions for bulk operations

    Best Practices¶

    Write Efficient Queries from the Start¶

    • Think about how your query will be translated to SQL
    • Consider the data volume you're working with
    • Profile queries during development, not just in production

    Balance Database Load and Application Complexity¶

    • Sometimes it's better to do multiple simple queries than one complex one
    • Consider the trade-off between query complexity and application logic
    • Don't over-optimize queries that are rarely executed

    Monitor in Production¶

    • Set up query logging in production (at a reasonable level)
    • Monitor slow query logs
    • Use APM tools to track database performance

    Troubleshooting Performance Issues¶

    Identifying Slow Queries¶

    1. Enable query logging
    2. Look for repeated patterns
    3. Check for missing indexes
    4. Analyze query execution plans

    Fixing N+1 Query Issues¶

    1. Identify the relationship causing the issue
    2. Determine if it's a single-valued (use select_related) or multi-valued (use prefetch_related) relationship
    3. Implement the appropriate optimization
    4. Verify with logging that the issue is resolved

    Resolving Lock Contention¶

    • Keep transactions short
    • Access tables in consistent order
    • Use appropriate isolation levels
    • Implement retry logic for deadlock-prone operations
